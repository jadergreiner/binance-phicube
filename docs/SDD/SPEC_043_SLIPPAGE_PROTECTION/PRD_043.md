## 1. Problema

### 1.1 A Dor

O bot Phicube apresenta **erosão silenciosa de capital por slippage não contabilizada na execução de stops**. Análise forense de 10 trades reais demonstra:

| Métrica                | Valor                        |
| ---------------------- | ---------------------------- |
| Win rate               | 70% (7/10)                   |
| Lucro médio (winners)  | +$2,72                       |
| Perda média (losers)   | −$8,35 (**3,1× o lucro**)        |
| **PnL líquido**            | **−$5,98**                       |
| Outlier CHZUSDT SHORT  | −$16,42 (**3,28×** risco $5,00) |

**70% de acerto deveria gerar lucro, mas gera prejuízo.** A estratégia BO Williams tem assimetria positiva teórica (win rate elevado), mas a infraestrutura de execução — especificamente o slippage em STOP_MARKET — anula essa assimetria. A cada 3 acertos, 1 perda descontrolada leva todo o ganho acumulado.

### 1.2 Causa Raiz

A perda de −$16,42 em CHZUSDT excedeu em $11,42 o risco configurado de $5,00. Análise identificou:

1. **Trigger por last price (padrão)** — wick momentâneo no livro de ofertas pode disparar o stop antes do preço justo confirmar
2. **Sem modelo de slippage na validação de risco** — o RiskManager aprova o trade baseado em risco teórico, mas o risco real (com slippage) pode ser 3× maior
3. **Sem circuit breaker** — após uma perda severa, o próximo trade opera com risco cheio novamente
4. **Sem limite dinâmico de posição** — `MAX_POSITION_USDT` fixo não escala com o saldo disponível

### 1.3 Evidências

- `docs/SDD/SPEC_043_SLIPPAGE_PROTECTION/SPEC.md` §1.2 — Análise forense completa
- Dashboard de 10 trades: 7 winners, 3 losers, net loss −$5,98
- `src/trading/risk_manager.py` — INV-029-05 só valida risco teórico, não considera slippage
- `src/exchange/binance_client.py` — STOP_MARKET sem priceProtect, sem parâmetro de slippage

## 2. Proposta de Valor

### 2.1 Promessa Central

> **Seu risco REAL é igual ao risco CONFIGURADO.**

| Antes (hoje)                                    | Depois (SPEC_043)                                                       |
| ----------------------------------------------- | ----------------------------------------------------------------------- |
| SL dispara por wick de last price               | SL trigger por **mark price** (preço justo) → priceProtect              |
| Perda real pode ser 3,28× o risco configurado   | Trade rejeitado se slippage estimada fizer risco total > 1,10× do configurado |
| Após perda severa, próximo trade opera com risco cheio | Circuit breaker reduz risco em **50%** após 3 perdas consecutivas por par |
| `MAX_POSITION_USDT` fixo ($500) não escala com saldo | `max_position_pct` escala automaticamente: posição limitada a % do saldo |
| Portfolio sem proteção contra correlação         | Portfolio circuit breaker reduz **25%** do risco global após 2 perdas em pares diferentes |

### 2.2 Alinhamento com o PRD Principal

| Princípio do PRD v1.2         | Como SPEC_043 entrega                                                     |
| ----------------------------- | ------------------------------------------------------------------------- |
| Disciplina                    | Regras de slippage e circuit breaker são automáticas, sem intervenção     |
| Gestão de risco               | Risco real ≤ risco configurado × 1,10. Stop loss com priceProtect.       |
| Transparência                 | Logs estruturados de `slippage_check_passed` e `circuit_breaker_activated` |
| Configurabilidade             | Thresholds, fatores de redução e tolerância de slippage configuráveis      |
| Segurança                     | priceProtect fixo (não falseável). Fallback com log WARN se exchange não suportar |

## 4. Decisões Arquiteturais (Time A + Arquiteto)

### 4.1 Decisões Aprovadas

| #   | Decisão                                              | Responsável          |
| --- | ---------------------------------------------------- | -------------------- |
| D01 | priceProtect adicionado com **fallback** para stops comuns (try/except + log WARN) | Backend Sênior       |
| D02 | Fallback do trailing stop: `except ccxt.BadRequest` (não `Exception` genérico) | Backend Sênior       |
| D03 | Circuit breaker pair-level: **threshold=3**, redução **50%**            | Risk Manager         |
| D04 | Circuit breaker portfólio: **threshold=2**, redução **25%**             | Risk Manager         |
| D05 | Circuit breaker **não persistido** em restart (INV-043-06 mantida)            | Backend Sênior       |
| D06 | Piso do circuit breaker: **$1,00** mantido                                 | Quant Developer      |
| D07 | Tolerância de slippage: **1,10×** normal, **1,05×** se risco reduzido pelo CB | Quant Developer      |
| D08 | Acoplamento OrderMonitor → RiskManager via **callback** `Callable[[str, float], None]` | Backend Sênior |
| D09 | `asyncio.Lock` no `register_trade_outcome()`                               | Backend Sênior       |
| D10 | PnL=0 tratado como **neutro** (zona morta entre -0,001 e +0,001)           | Quant Developer      |
| D11 | priceProtect lag documentado como limitação (sem alerta adicional)         | Backend Sênior       |

### 4.2 Arquitetura do Circuit Breaker (Dual-Level)

```
Pair-Level CB (RiskManager por par)          Portfolio CB (RiskManager central)
┌─────────────────────────────────────┐      ┌─────────────────────────────┐
│ threshold=3, reduction=0.5          │      │ threshold=2, reduction=0.75 │
│ register_trade_outcome(pnl)         │      │ register_portfolio_outcome  │
│ → perda consecutiva no símbolo      │      │ → perda em qualquer par     │
│ → ativa APÓS a 3ª perda             │      │ → ativa APÓS 2 perdas em    │
│ → reduz risk_per_trade_usdt do par  │      │   pares diferentes          │
│ → pair_breaker_active = True        │      │ → reduz risk global         │
└─────────────────────────────────────┘      └─────────────────────────────┘
         │                                              │
         └────────────── Ambas são verificadas ──────────┘
                              │
                              ▼
         effective_risk = min(pair_risk, portfolio_risk * portfolio_reduction)
```

### 4.3 Integração OrderMonitor → RiskManager

O `OrderMonitor` (singleton) recebe no construtor um callback:
```python
on_trade_closed: Callable[[str, float], None] | None = None
# Parâmetros: (symbol: str, pnl_usdt: float)
```

O `TradingMonitor` (que possui acesso aos `RiskManager`s por par) registra o callback:
```python
async def _on_trade_closed(self, symbol: str, pnl_usdt: float) -> None:
    risk_manager = self._risk_managers.get(symbol)
    if risk_manager:
        risk_manager.register_trade_outcome(pnl_usdt)
    # Portfolio CB: chamar MainRiskManager.register_portfolio_outcome(pnl_usdt)
```

Isso mantém o `OrderMonitor` desacoplado do `RiskManager` — ele só conhece o contrato `Callable`, não a classe concreta.

## 5. Personas

### Operador / Dono do Sistema

- **Necessidade:** Stops não disparam falsamente em wicks. Capital preservado em drawdowns sem intervenção manual.
- **Sucesso:** Pode dormir sabendo que o risco real de cada trade está dentro do configurado.
- **Preocupações:** "O bot vai deixar de entrar em trades bons por causa de um modelo de slippage conservador demais?"

### Gestor de Risco

- **Necessidade:** Slippage estimada antes de cada trade. Rejeição automática de pares de baixa liquidez com exposição grande.
- **Sucesso:** Toda perda fica dentro dos limites estabelecidos.

## 8. Riscos e Mitigações

| Risco                                                                                     | Impacto | Mitigação                                                         | Resolução                                       |
| ----------------------------------------------------------------------------------------- | ------- | ----------------------------------------------------------------- | ----------------------------------------------- |
| priceProtect pode atrasar execução em flash crash (mark price lag)                        | Médio   | BO Williams opera em 15m+ — lag de 2-3s é irrelevante para a estratégia | Aceito (D11)                                    |
| Slippage estimada (tiers estáticos) pode ser imprecisa em eventos de volatilidade extrema | Médio   | Slippage dinâmico via ATR captura mudanças parciais; tiers são conservadores (low=0,15%) | Mitigado por R2+R3                              |
| Circuit breaker pode reduzir exposição durante sequência legítima de perdas               | Baixo   | Threshold=3 permite 2 perdas isoladas. Redução de 50% ainda permite operar | Aceito (D03)                                    |
| Portfolio breaker ativa com apenas 2 perdas em pares diferentes — pode ser sensível demais | Baixo   | Redução é de 25% apenas. Pair-level ainda protege por par com 50% após 3 | Aceito (D04)                                    |
| Contador do CB não persistido — bypassável por restart                                    | Baixo   | Restart é evento raro em produção. Log de restart + auditoria detectam abuso | Aceito (D05)                                    |
| `max_position_pct=0` pode travar todos os trades                                         | Baixo   | Validação de startup + doc clara no .env.example                       | Mitigado por validação                          |

## 11. Glossário

| Termo                       | Definição                                               |
| --------------------------- | ------------------------------------------------------- |
| priceProtect                | Parâmetro da Binance que faz o trigger usar mark price  |
| Slippage estimada           | Custo projetado de slippage = `max(tier_pct, ATR%×0.5) × notional` |
| Circuit breaker             | Redução automática de risco após N perdas consecutivas  |
| Tolerância de slippage      | `(risco + slippage) / risco_configurado` — default 1,10× |
| max_position_pct            | Limite de posição como % do saldo (adicional ao MAX_POSITION_USDT) |
| Pair-level                  | Por par (cada símbolo tem seu RiskManager)              |
| Portfolio-level             | Global (compartilhado entre todos os pares)              |
